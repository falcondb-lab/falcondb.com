# 为什么"报工成功"不能丢？

# Why "Confirmed Report" Must Never Be Lost

本文面向工厂管理层、制造业 IT 团队、MES 评估人员。
不使用数据库术语。用制造业的语言解释系统行为。

---

## 场景：一个真实的生产事故

```
周二 14:23
  操作员张工完成了 200 个零件的加工。
  系统显示：报工成功。
  张工继续去做下一批。

周二 14:25
  机房突然断电。
  服务器宕机。

周二 14:40
  电力恢复，系统重启。

问题来了：
  张工报的那 200 个零件，还在不在？
```

**如果答案是"不在"——**
- 张工要重新手动补报
- 如果他忘了，200 个零件就从系统里消失了
- 月底盘点对不上帐
- 审计时无法解释差异

**如果答案是"还在"——**
- 张工不需要做任何事
- 生产记录完整
- 审计可追溯
- 系统值得信任

## 这个系统保证什么？

### 核心承诺

> **系统一旦返回"报工成功"，这条生产记录就永远不会因为任何系统故障而消失。**

不管发生什么：
- 服务器突然断电 ✅ 记录不丢
- 主机宕机 ✅ 记录不丢
- 数据库进程被异常终止 ✅ 记录不丢
- 网络中断后恢复 ✅ 记录不丢

### 用生活类比

想象一下银行转账：

```
你在 ATM 上转了 5000 元。
屏幕显示：转账成功。

然后银行的机房停电了。

你打电话去问：我的 5000 元还在吗？

银行回答：当然在。我们已经确认了这笔交易。
          机房停电不影响已确认的交易。
```

**我们的 MES 系统做了同样的事情，但用在了生产报工上。**

## 什么是工单？

### 用大白话说

工单 = 一张生产任务单。

```
工单 #1234
  产品：电机 A100
  计划数量：1000 台
  状态：生产中

  工序 1：下料    → 已完成
  工序 2：焊接    → 已完成
  工序 3：组装    → 进行中（已报工 600 台）
  工序 4：质检    → 等待中
```

### 工单的生命周期

```
已创建 → 生产中 → 已完成

  CREATED → IN_PROGRESS → COMPLETED

  状态只能往前走，不能往回退。
  一旦标记为"已完成"，就不能再改回"生产中"。
```

## 什么是报工？

### 用大白话说

报工 = 操作员告诉系统："我完成了多少个"。

```
报工记录 #5678
  工单：#1234
  工序：组装
  数量：200 台
  操作员：张工
  时间：2026-03-01 14:23:15
```

### 报工的规则

1. **只能新增，不能修改，不能删除**
   - 每一条报工记录 = 一个不可否认的生产事实
   - 就像会计账本：只记正数，错了就用红字冲减

2. **数量只能增加，不能减少**
   - 如果张工报了 200，李工报了 150
   - 累计就是 350，永远不会变成 300

3. **报工必须确认后才返回成功**
   - 系统不会说"我先告诉你成功，后面再存"
   - 而是"我确实存好了，然后才告诉你成功"

## 宕机时发生了什么？

### 不用技术术语的解释

```
正常运行时：
  ┌──────────┐     ┌──────────┐
  │  操作员   │ ──→ │  系统     │ ──→ "报工成功"
  │  报工     │     │  记录     │
  └──────────┘     └──────────┘

系统确认的方式：
  1. 操作员提交报工
  2. 系统把这条记录写入"永久记忆"
  3. 系统确认"永久记忆"已写入
  4. 系统才返回"报工成功"

宕机时：
  ┌──────────┐     ┌──────────┐
  │  操作员   │     │  系统宕机  │
  └──────────┘     └──────────┘

恢复时：
  系统重新启动
  ↓
  读取"永久记忆"
  ↓
  恢复所有已确认的记录
  ↓
  系统回到宕机前的状态

  结果：已确认的报工记录一条都没丢。
```

### 关键点

**"永久记忆"在系统说"成功"之前就已经写好了。**

所以即使系统在说完"成功"后一毫秒就宕机了，
那条记录仍然安全。

## 为什么传统 MES 做不到这一点？

### 常见的错误做法

| 做法 | 问题 |
|------|------|
| 先返回成功，后台异步存储 | 宕机时数据可能还没存完 |
| 用缓存加速，定期写入数据库 | 缓存在内存里，断电就没了 |
| 存在应用服务器内存里 | 服务器重启数据全丢 |
| 数据库没有开启同步确认 | 数据库说"存好了"但其实还在缓冲区 |

### 我们的做法

| 做法 | 保证 |
|------|------|
| 每次报工都是独立事务 | 要么完整存入，要么完全不存 |
| 同步确认后才返回 | 系统说"成功"时数据已在磁盘 |
| 没有应用层缓存 | 不存在"缓存丢失"的可能 |
| 数据库内置恢复机制 | 宕机后自动恢复所有已确认数据 |

## 对审计意味着什么？

### 审计人员关心的问题

| 问题 | 回答 |
|------|------|
| 生产数据能丢吗？ | 已确认的不能丢 |
| 报工能被篡改吗？ | 不能，只能新增 |
| 宕机后数据还在吗？ | 在，一条不少 |
| 能证明吗？ | 能，运行验证脚本 |
| 工序能跳过吗？ | 不能，必须按顺序 |
| 工单能倒退吗？ | 不能，状态单向前进 |

### 合规覆盖

- **GMP（药品生产规范）**: 生产记录不可篡改 ✅
- **ISO 9001**: 可追溯的质量记录 ✅
- **汽车行业 IATF 16949**: 过程控制记录完整 ✅
- **食品安全 HACCP**: 关键控制点记录可靠 ✅
