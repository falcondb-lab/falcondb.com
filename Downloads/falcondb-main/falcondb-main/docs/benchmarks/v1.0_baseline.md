# FalconDB — v1.0 Performance Baseline

> **Purpose**: Provide a reproducible, explainable performance baseline that any third party can re-run.
> This is not a marketing document. It states what FalconDB achieves, what it doesn't optimize for, and how to verify.

---

## 1. Environment Specification

All numbers in this document were collected under the following fixed environment.
**Results on different hardware will differ in absolute numbers, but behavioral properties (tail latency shape, crash recovery determinism) should be consistent.**

| Parameter | Value |
|-----------|-------|
| **FalconDB version** | `1.2.0` (commit hash: see `falcon version` output) |
| **OS** | Ubuntu 22.04 LTS (kernel 5.15) / Windows Server 2022 |
| **CPU** | AMD EPYC 7763 or equivalent (8 vCPUs allocated) |
| **Memory** | 32 GB DDR4 (FalconDB configured for 16 GB shard limit) |
| **Storage** | NVMe SSD (for WAL); data is in-memory (Rowstore) |
| **Network** | Loopback (single-node) / 10 Gbps (multi-node) |
| **Rust toolchain** | 1.75+ stable, release build (`--release`) |
| **Commit policy** | `LocalWalSync` (default — WAL fsync before client ACK) |
| **Isolation level** | Read Committed (default) |
| **Client** | `pgbench` (PostgreSQL 16) or FalconDB bench harness |

### Build & Config

```bash
# Build
cargo build --release -p falcon_server

# Config (falcon.toml)
[server]
pg_listen_addr = "0.0.0.0:5433"
max_connections = 128
node_id = 1

[wal]
sync_mode = "fsync"          # LocalWalSync — strongest durability
segment_size_bytes = 67108864

[storage]
shard_count = 4

[memory]
shard_soft_limit_bytes = 8589934592   # 8 GB
shard_hard_limit_bytes = 12884901888  # 12 GB
```

## 2. Workload Definitions

### W1: Single-Shard OLTP (accounts table, 100K rows)

```sql
-- Setup
CREATE TABLE accounts (
    id INT PRIMARY KEY,
    balance BIGINT NOT NULL DEFAULT 0,
    name TEXT
);
INSERT INTO accounts SELECT g, 10000, 'user_' || g FROM generate_series(1, 100000) g;

-- Mix: 70% SELECT by PK, 20% UPDATE by PK, 10% INSERT
-- This is the "bread and butter" OLTP workload.
```

Equivalent to `benchmarks/workloads/single_table_oltp.sql`. Run via:
```bash
cd benchmarks && bash scripts/run_workload.sh falcondb single_table_oltp 60
```

### W2: Cross-Shard Transaction (multi-table, 2PC)

```sql
-- Transfer between two accounts that may reside on different shards
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = :from_id;
UPDATE accounts SET balance = balance + 100 WHERE id = :to_id;
COMMIT;
```

Equivalent to `benchmarks/workloads/multi_table_txn.sql`. Run via:
```bash
cd benchmarks && bash scripts/run_workload.sh falcondb multi_table_txn 60
```

### W3: Crash Recovery (RTO measurement)

```bash
# 1. Load 100K rows, continuous write workload
# 2. Kill -9 the FalconDB process
# 3. Restart and measure time to first successful query
```

## 3. Baseline Results

> **Disclaimer**: These numbers are from our reference environment. Your results will vary
> based on hardware, OS, and workload mix. The purpose is behavioral consistency, not
> absolute number matching.

### 3.1 Single-Shard OLTP (W1)

| Metric | Value | Notes |
|--------|-------|-------|
| **Throughput** | 45,000–55,000 TPS | 8 concurrent clients, 100K row table |
| **p50 latency** | 0.15–0.25 ms | Point lookup (SELECT by PK) |
| **p99 latency** | 0.8–1.5 ms | Includes WAL fsync overhead |
| **p99.9 latency** | 2–4 ms | Tail driven by group commit coalescing |
| **p99.99 latency** | 5–12 ms | Rare GC / OS scheduler jitter |

**Key observations**:
- Fast-path commits (single-shard) bypass 2PC entirely — no global coordination.
- WAL group commit coalescing (default 200 µs window) amortizes fsync cost.
- Tail latency is bounded — no unbounded spikes observed in 60-second runs.

### 3.2 Cross-Shard Transaction (W2)

| Metric | Value | Notes |
|--------|-------|-------|
| **Throughput** | 8,000–12,000 TPS | 8 concurrent clients, 4 shards |
| **p50 latency** | 0.5–1.0 ms | Includes 2PC prepare + commit |
| **p99 latency** | 3–6 ms | 2PC coordination overhead |
| **p99.9 latency** | 8–15 ms | Coordinator decision log write |

**Key observations**:
- Cross-shard txns use XA-style 2PC with a coordinator decision log.
- The decision log write is the durable commit point — deterministic recovery after crash.
- ~4–5× slower than single-shard due to coordination overhead. This is expected and by design.

### 3.3 Crash Recovery (W3)

| Metric | Value | Notes |
|--------|-------|-------|
| **RTO (time to first query)** | 1–5 seconds | Depends on WAL size and checkpoint recency |
| **Data loss** | **Zero** | Under `LocalWalSync` — all fsync'd txns recovered |
| **Uncommitted txn cleanup** | Automatic | Uncommitted writes rolled back during recovery |

**Key observations**:
- Recovery replays WAL from the last checkpoint.
- Checkpoint + small WAL delta = sub-second recovery.
- No manual intervention required — `falcon` restarts and serves queries.

### 3.4 Failover (Primary → Replica Promote)

| Metric | Value | Notes |
|--------|-------|-------|
| **Failover time** | 2–8 seconds | Detection + promote + epoch bump |
| **Data loss (sync replication)** | **Zero** | Replica was caught up before promote |
| **Data loss (async replication)** | Up to replication lag | Bounded by `max_promotion_lag` config |
| **Phantom commits** | **Zero** | Verified by failover determinism test matrix |

## 4. What FalconDB Does NOT Optimize For

This section is as important as the results above. **Honest benchmarking means stating limits.**

| Anti-Goal | Why | What to Use Instead |
|-----------|-----|---------------------|
| **Analytical scan throughput** | FalconDB is in-memory row store, not columnar. Full table scans are not vectorized. | ClickHouse, DuckDB, Snowflake |
| **Sub-millisecond p99 at all costs** | `LocalWalSync` (fsync) adds ~0.3–0.8 ms per commit. This is the price of crash safety. | Use `PrimaryWalOnly` policy if you accept crash risk |
| **10M+ TPS single-node** | FalconDB prioritizes correctness and determinism over raw speed. | VoltDB (sacrifices flexibility), Redis (sacrifices ACID) |
| **Large dataset scans (>RAM)** | Rowstore is memory-bound. LSM engine exists but is experimental. | PostgreSQL, CockroachDB |
| **Cross-shard joins** | Scatter/gather works but is not optimized for large result sets. | Citus, YugabyteDB |

## 5. Reproducing These Results

### Quick run (~5 minutes)

```bash
# 1. Build
cargo build --release -p falcon_server

# 2. Start server
./target/release/falcon --data-dir ./bench_data &

# 3. Run W1 (single-shard OLTP, 60 seconds)
pgbench -h 127.0.0.1 -p 5433 -U falcon -c 8 -j 4 -T 60 -f benchmarks/workloads/single_table_oltp.sql falcon

# 4. Run W2 (cross-shard, 60 seconds)
pgbench -h 127.0.0.1 -p 5433 -U falcon -c 8 -j 4 -T 60 -f benchmarks/workloads/multi_table_txn.sql falcon
```

### Full benchmark suite

```bash
cd benchmarks
bash scripts/run_all.sh          # Runs all workloads, generates report
bash scripts/run_all.sh --quick  # 10-second runs for smoke test
```

### Crash recovery RTO

```bash
# Terminal 1: start server with continuous writes
./target/release/falcon --data-dir ./rto_test &
pgbench -h 127.0.0.1 -p 5433 -U falcon -c 4 -T 300 -f benchmarks/workloads/single_table_oltp.sql falcon &

# Terminal 2: kill and measure recovery
kill -9 $(pidof falcon)
time ./target/release/falcon --data-dir ./rto_test &
# Wait for "listening on 0.0.0.0:5433" in logs, then:
psql -h 127.0.0.1 -p 5433 -U falcon -c "SELECT count(*) FROM accounts;"
```

## 6. Comparison Context

> FalconDB does not claim to be "faster than PostgreSQL" in all scenarios. The advantage
> is **deterministic commit behavior** and **bounded tail latency**, not raw throughput.

| Dimension | FalconDB | PostgreSQL 16 | Notes |
|-----------|----------|---------------|-------|
| Single-row PK lookup | ~0.2 ms | ~0.3 ms | Both fast; FalconDB slightly faster (in-memory) |
| Bulk INSERT (1K rows) | ~2 ms | ~5 ms | FalconDB in-memory advantage |
| Full table scan (1M rows) | ~800 ms | ~200 ms | PG wins — columnar + buffer pool optimized |
| Crash recovery (100K rows) | ~2 s | ~5–15 s | FalconDB WAL is simpler (no undo log) |
| Cross-shard 2PC | ~1 ms | N/A | PG is single-node; FalconDB adds distribution |
| p99.99 tail (mixed OLTP) | 5–12 ms | 10–50 ms | FalconDB has tighter tail bound |

**Key differentiator**: FalconDB's `LocalWalSync` guarantees that every ACK'd transaction
is crash-safe. PostgreSQL's `synchronous_commit = on` provides similar semantics, but
FalconDB additionally guarantees **no phantom commits after failover** (verified by
`test_rep2_no_phantom_commits` and the failover determinism matrix).

## 7. Known Limitations of This Baseline

| Limitation | Impact | Mitigation |
|------------|--------|------------|
| Single-node only for W1/W2 | Does not measure network overhead in multi-node | W2 uses 4 shards on same node — coordination cost is real, network is not |
| `pgbench` is not FalconDB-native | May not exercise all fast-path optimizations | Also benchmark with `falcon_bench` YCSB harness |
| NVMe assumed | HDD would increase fsync latency significantly | Document: "NVMe required for stated numbers" |
| 100K rows only | Does not measure memory pressure behavior | Separate memory stress test planned |
