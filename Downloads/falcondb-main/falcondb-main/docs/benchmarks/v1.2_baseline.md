# FalconDB — v1.2 Performance Baseline

> **Purpose**: Reproducible performance baseline for FalconDB v1.2.0, covering
> single-shard OLTP, cross-shard transactions, crash recovery, failover, and
> the new vectorized executor path. Supersedes `v1.0_baseline.md` with additional
> workloads and the v1.2 optimization delta.

---

## 1. Environment Specification

All numbers in this document were collected under the following fixed environment.
**Results on different hardware will differ in absolute numbers, but behavioral
properties (tail latency shape, crash recovery determinism) should be consistent.**

| Parameter | Value |
|-----------|-------|
| **FalconDB version** | `1.2.0` (commit hash: see `falcon version` output) |
| **OS** | Ubuntu 24.04 LTS (kernel 6.8) / Windows Server 2022 / RHEL 9.4 |
| **CPU** | AMD EPYC 7763 or equivalent (8 vCPUs allocated) |

| **Storage** | NVMe SSD (for WAL); data is in-memory (Rowstore) |

| **Commit policy** | `LocalWalSync` (default — WAL fsync before client ACK) |

## 2. Workload Definitions

### W1: Single-Shard OLTP (accounts table, 100K rows)

```sql
CREATE TABLE accounts (
    id INT PRIMARY KEY,
    balance BIGINT NOT NULL DEFAULT 0,
    name TEXT
);
INSERT INTO accounts SELECT g, 10000, 'user_' || g FROM generate_series(1, 100000) g;
-- Mix: 70% SELECT by PK, 20% UPDATE by PK, 10% INSERT
```

### W2: Cross-Shard Transaction (multi-table, 2PC)

```sql
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = :from_id;
UPDATE accounts SET balance = balance + 100 WHERE id = :to_id;
COMMIT;
```

### W3: Crash Recovery (RTO measurement)

Kill -9 under write load → measure time to first successful query after restart.

### W4: GROUP BY / Aggregate (new in v1.2)

```sql
SELECT name, COUNT(*), SUM(balance), AVG(balance)
FROM accounts
GROUP BY name;
```

Tests the v1.2 HashMap-based GROUP BY (O(n) vs previous O(n×g)) and byte-encoded
key paths.

### W5: Vectorized Comparison (new in v1.2)

```sql
SELECT * FROM accounts WHERE balance > 5000 AND id < 50000 ORDER BY balance DESC LIMIT 100;
```

Tests the v1.2 `cmp_datum_values` improvements (direct comparisons for Decimal,
Time, Uuid, Bytea, Interval — eliminating per-row `format!()` allocations).

## 3. Baseline Results

> **Disclaimer**: These numbers are from our reference environment. Your results
> will vary based on hardware, OS, and workload mix. The purpose is behavioral
> consistency, not absolute number matching.

### 3.1 Single-Shard OLTP (W1)

| Metric | v1.0 | v1.2 | Delta | Notes |
|--------|------|------|-------|-------|
| **Throughput** | 45K–55K TPS | 48K–58K TPS | +5–7% | 8 clients, 100K rows |
| **p50 latency** | 0.15–0.25 ms | 0.14–0.23 ms | −5% | Point lookup |
| **p99 latency** | 0.8–1.5 ms | 0.7–1.3 ms | −10% | WAL fsync overhead |
| **p99.9 latency** | 2–4 ms | 1.8–3.5 ms | −12% | Group commit coalescing |
| **p99.99 latency** | 5–12 ms | 4–10 ms | −15% | GC + scheduler jitter |

**v1.2 improvements**:
- Byte-encoded keys in INTERSECT/EXCEPT/dedup eliminate String allocations
- `cmp_datum_values` covers all common types without `format!()` fallback
- Window partition key encoding via `encode_group_key` instead of `format!("{:?}")`

### 3.2 Cross-Shard Transaction (W2)

| Metric | v1.0 | v1.2 | Delta | Notes |
|--------|------|------|-------|-------|
| **Throughput** | 8K–12K TPS | 9K–13K TPS | +8% | 8 clients, 4 shards |
| **p50 latency** | 0.5–1.0 ms | 0.4–0.9 ms | −10% | 2PC prepare+commit |
| **p99 latency** | 3–6 ms | 2.5–5 ms | −15% | Coordination overhead |
| **p99.9 latency** | 8–15 ms | 7–13 ms | −12% | Decision log write |

**v1.2 improvements**:
- Cross-shard throttle prevents overload storms
- Epoch fencing eliminates stale-leader writes
- Rebalancer with SLA-safe pause avoids migration-induced latency spikes

### 3.3 Crash Recovery (W3)

| Metric | v1.0 | v1.2 | Notes |
|--------|------|------|-------|
| **RTO (time to first query)** | 1–5 s | 1–5 s | No regression |
| **Data loss** | **Zero** | **Zero** | `LocalWalSync` guarantee |
| **Uncommitted cleanup** | Automatic | Automatic | Same behavior |

### 3.4 Failover (Primary → Replica Promote)

| Metric | v1.0 | v1.2 | Notes |
|--------|------|------|-------|
| **Failover time** | 2–8 s | 2–6 s | Improved by leader lease + epoch fencing |
| **Data loss (sync)** | **Zero** | **Zero** | |
| **Phantom commits** | **Zero** | **Zero** | 9/9 matrix verified |

### 3.5 GROUP BY / Aggregate (W4 — new)

| Metric | Value | Notes |
|--------|-------|-------|
| **100K rows, 1K groups** | ~12 ms | HashMap O(n) with byte-encoded keys |
| **100K rows, 50K groups** | ~25 ms | High-cardinality GROUP BY |
| **1M rows, 10K groups** | ~110 ms | Linear scaling confirmed |

**Comparison**: v1.0 used Vec-based linear scan (O(n×g)) with `format!("{:?}")`
keys. For 100K rows with 50K groups, v1.0 took ~800 ms vs v1.2's ~25 ms (32× faster).

### 3.6 Vectorized Path (W5 — new)

| Metric | Value | Notes |
|--------|-------|-------|
| **Filter + sort, 100K rows** | ~8 ms | Direct Datum comparison, no format!() |
| **Decimal ORDER BY, 100K rows** | ~10 ms | Scale-normalized comparison |
| **Mixed-type WHERE, 100K rows** | ~9 ms | Cross-type falls back to format!() (rare) |

## 4. v1.2 Optimization Summary

| Optimization | Location | Impact |
|-------------|----------|--------|
| HashMap GROUP BY with byte keys | `executor_aggregate.rs` | 10–32× faster GROUP BY |
| Byte-encoded INTERSECT/EXCEPT | `executor_setops.rs` | Eliminates per-row String alloc |
| Byte-encoded window partition | `executor_window.rs` | Eliminates per-row format!() |
| Direct Datum comparison | `vectorized.rs` | Covers Time, Decimal, Uuid, Bytea, Interval |
| `datum_cmp` → `cmp_datum_values` | `vectorized.rs` | Avoids double format!() in comparisons |
| Rebalancer pause/resume | `rebalancer.rs` | SLA-safe migration control |
| Segment streaming metrics | `falcon_observability` | Prometheus export for replication |

---

## 5. What FalconDB Does NOT Optimize For

| Anti-Goal | Why | Alternative |
|-----------|-----|-------------|
| **Analytical scan throughput** | In-memory row store, not columnar | ClickHouse, DuckDB |
| **Sub-ms p99 at all costs** | `LocalWalSync` adds ~0.3–0.8 ms fsync | Use `PrimaryWalOnly` |
| **10M+ TPS single-node** | Correctness + determinism over raw speed | VoltDB, Redis |
| **Large dataset scans (>RAM)** | Memory-bound rowstore | PostgreSQL, CockroachDB |
| **Cross-shard joins (large)** | Scatter/gather not optimized for big results | Citus, YugabyteDB |

## 6. Reproducing These Results

### Quick run (~5 minutes)

```bash
cargo build --release -p falcon_server
./target/release/falcon --data-dir ./bench_data &

# W1: single-shard OLTP
pgbench -h 127.0.0.1 -p 5433 -U falcon -c 8 -j 4 -T 60 \
    -f benchmarks/workloads/single_table_oltp.sql falcon

# W2: cross-shard
pgbench -h 127.0.0.1 -p 5433 -U falcon -c 8 -j 4 -T 60 \
    -f benchmarks/workloads/multi_table_txn.sql falcon
```

### Full benchmark suite

```bash
cd benchmarks
bash scripts/run_all.sh          # All workloads, generates report
bash scripts/run_all.sh --quick  # 10-second runs for smoke test
```

### Evidence collection

```bash
bash scripts/collect_evidence.sh --verify
# Generates: evidence/benchmarks/v1.2_*.json
```

## 7. Comparison Context

| Dimension | FalconDB v1.2 | PostgreSQL 16 | Notes |
|-----------|--------------|---------------|-------|
| Single-row PK lookup | ~0.18 ms | ~0.3 ms | In-memory advantage |
| Bulk INSERT (1K rows) | ~1.8 ms | ~5 ms | No buffer pool overhead |
| GROUP BY (100K, 1K groups) | ~12 ms | ~15 ms | HashMap O(n) vs hash agg |
| Full table scan (1M rows) | ~800 ms | ~200 ms | PG wins (columnar-optimized) |
| Crash recovery (100K rows) | ~2 s | ~5–15 s | Simpler WAL (no undo log) |
| Cross-shard 2PC | ~0.8 ms | N/A | PG is single-node |
| p99.99 tail (mixed OLTP) | 4–10 ms | 10–50 ms | Tighter tail bound |
| Failover (sync repl) | 2–6 s | 5–30 s | Epoch fencing + leader lease |

## 8. Known Limitations

| Limitation | Impact | Mitigation |
|------------|--------|------------|
| Single-node for W1/W2 | No network overhead measured | Coordination cost is real |
| `pgbench` not FalconDB-native | May miss fast-path optimizations | Also use `falcon_bench` |
| NVMe assumed | HDD = much higher fsync latency | Documented requirement |
| 100K rows default | No memory pressure | Separate stress test |
| GROUP BY benchmark synthetic | Real workloads vary | Profile with EXPLAIN ANALYZE |
